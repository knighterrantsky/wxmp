# 真机手工验收清单

## 1. 执行规则

本清单必须在发布前使用真实微信账号、真机、真实 `wx.login -> code2Session`、HTTPS 测试 API、独立测试 PostgreSQL 与独立私有 R2 bucket 完成。不得在生产 bucket 做破坏性故障注入。

开始前必须在微信公众平台《小程序用户隐私保护指引》中如实声明昵称收集及用途；否则 `<input type="nickname">` 不会提供微信昵称候选，本节验收没有意义。验收设备必须支持微信基础库 2.32.3 或更高版本；更低版本应在点击授权入口时明确提示升级，不能进入一个无法完成的授权流程。

记录以下信息：

| 项目                   | 结果 |
| ---------------------- | ---- |
| 小程序版本/commit      |      |
| API 镜像 tag/commit    |      |
| iOS/Android 机型与系统 |      |
| 微信版本               |      |
| 微信基础库版本         |      |
| 测试环境与 API 域名    |      |
| 测试 bucket            |      |
| 执行人/日期            |      |

每项填写 `通过`、`失败` 或 `阻塞`，附响应 `requestId`、截图时间和内部测试数据 ID。不要记录 access token、refresh token、AppSecret、R2 凭据、数据库 URL 或原始用户 openid。

## 2. 登录、稳定用户 ID 与昵称授权

### 2.1 首次与再次启动

1. 清除小程序本地数据，首次打开上传页。
2. 通过服务端日志/requestId 确认发生真实 `wx.login -> code2Session`，不是 stub。
3. 记录 API 返回的公开内部 `user.id`，确认 `nickname=null`、`nicknameConfirmed=false`。
4. 完全退出并重新打开小程序，再次记录 `user.id`。
5. 预期两次内部 ID 相同，客户端从未收到 openid、unionid 或 `session_key`。

| 用例           | 预期                              | 结果/证据 |
| -------------- | --------------------------------- | --------- |
| 首次真实登录   | 创建或映射一个内部用户 ID         |           |
| 再次启动       | 返回相同内部用户 ID               |           |
| 客户端数据检查 | 无微信身份底层标识和服务端 secret |           |

### 2.2 昵称候选与显式确认

1. 未同意昵称隐私授权时点击昵称区域，确认出现授权申请与隐私指引入口。
2. 拒绝或关闭授权，确认没有保存昵称，上传初始化仍被阻止。
3. 同意后聚焦真实 `<input type="nickname">`，确认微信展示昵称候选。
4. 选择候选但离开/取消独立确认动作，重新打开页面；预期服务端仍是 `nickname=null`、`nicknameConfirmed=false`。
5. 再次选择候选，点击页面上单独、可见的确认动作。
6. 重新启动小程序；预期昵称与 `nicknameConfirmed=true` 持久存在。
7. 在验收记录中明确：该昵称只是用户确认资料，微信昵称组件不提供服务端可验证身份签名，不能写成“微信认证身份”。

| 用例         | 预期                   | 结果/证据 |
| ------------ | ---------------------- | --------- |
| 拒绝隐私授权 | 不保存昵称，上传仍阻止 |           |
| 候选弹出     | 微信昵称候选可见       |           |
| 选择后不确认 | 服务端不保存           |           |
| 单独确认     | 保存并跨重启保持       |           |
| 身份表述     | 仅为用户确认资料       |           |

## 3. 文件边界

所有大小按字节构造，不能用文件系统四舍五入后的显示值替代。有效样本的扩展名、MIME 和 magic bytes 必须一致。

| 样本                   |                 字节数 | 预期                                 | 结果/证据 |
| ---------------------- | ---------------------: | ------------------------------------ | --------- |
| 过小文件               |                     11 | 客户端拒绝；不得初始化上传           |
| 各格式最小有效签名样本 |                至少 12 | 通过本地检查；首片通过服务端签名检查 |
| 小文件                 |     5,242,880（5 MiB） | 1 片，完成后 `uploaded`              |
| 分片边界               |     8,388,608（8 MiB） | 1 片，大小精确                       |
| 跨分片边界             |              8,388,609 | 2 片，第二片 1 byte                  |
| 大文件                 | 208,666,624（199 MiB） | 25 片，完成上传                      |
| 最大合法文件           | 209,715,200（200 MiB） | 25 片，完成上传并显示真实进度        |
| 超限文件               |            209,715,201 | 客户端拒绝；不得初始化上传           |

对 11 bytes、200 MiB + 1 byte 和其他客户端拒绝项，检查数据库和测试 bucket，确认没有新上传会话或可用完整对象。

## 4. 格式白名单

分别准备一个合法文件和一个“扩展名/MIME 与 magic bytes 不一致”的反例：

| 类型    | 合法扩展名/MIME              | 合法结果   | 反例结果                  |
| ------- | ---------------------------- | ---------- | ------------------------- |
| JPEG    | `.jpg`/`.jpeg`；`image/jpeg` | `uploaded` | `MIME_MISMATCH`，安全终止 |
| PNG     | `.png`；`image/png`          | `uploaded` | 同上                      |
| WebP    | `.webp`；`image/webp`        | `uploaded` | 同上                      |
| GIF     | `.gif`；`image/gif`          | `uploaded` | 同上                      |
| HEIC    | `.heic`；`image/heic`        | `uploaded` | 同上                      |
| HEIF    | `.heif`；`image/heif`        | `uploaded` | 同上                      |
| MP4/M4V | `.mp4`/`.m4v`；`video/mp4`   | `uploaded` | 同上                      |
| MOV     | `.mov`；`video/quicktime`    | `uploaded` | 同上                      |

额外选择 SVG、HTML、脚本、压缩包、音频、可执行文件和 `application/octet-stream`，预期在初始化前或初始化时被拒绝。格式签名失败后历史最终应显示安全失败状态，不能停留为可继续写入。

## 5. 二次确认与上传进度

1. 选择 2 个合法文件，检查确认框显示准确的文件数和总大小。
2. 点击“取消”；确认未发起 `POST /v1/uploads`，数据库与 bucket 均无新增。
3. 重新选择并点击“开始上传”；确认此后才初始化上传。
4. 使用 8 MiB + 1 byte 和 200 MiB 文件观察进度：不得倒退、超过 100% 或在服务端确认前计为完成。
5. 最后一片完成后应先显示 `finalizing`，只有服务端查询返回终态后才显示 `uploaded`。
6. 打开上传记录页，确认显示文件名、大小、进度/状态与时间，不显示存储路径、multipart 标识或下载能力。

| 用例         | 预期                           | 结果/证据 |
| ------------ | ------------------------------ | --------- |
| 二次确认取消 | 无上传会话和存储写入           |           |
| 二次确认接受 | 初始化并开始分片               |           |
| 并发进度     | 单调、0–100%、以确认字节为准   |           |
| 完成状态     | `finalizing` 后才到 `uploaded` |           |
| 历史记录     | 仅安全业务字段                 |           |

## 6. 网络与令牌续期

每种场景使用至少 8 MiB + 1 byte 文件并记录已确认 part number：

| 场景         | 操作                                 | 预期                                                   | 结果/证据 |
| ------------ | ------------------------------------ | ------------------------------------------------------ | --------- |
| Wi-Fi 切蜂窝 | 上传中关闭 Wi-Fi                     | 当前请求可失败并按相同 part 重试；进度不倒退           |
| 蜂窝切 Wi-Fi | 上传中恢复 Wi-Fi                     | 只继续缺失分片                                         |
| 弱网         | 限速并制造抖动                       | 有界退避，不生成重复上传会话                           |
| 离线         | 上传中断网后恢复                     | 不把失败分片计为确认；恢复后查询服务端状态             |
| 超过 15 分钟 | 限速使传输跨过 access-token 生命周期 | 刷新令牌或重新登录后继续同一合法会话，不重传已确认分片 |

检查同一初始化/complete/abort 业务周期的网络重试复用原 UUIDv7 Idempotency-Key。普通超时不能生成新的 complete Key。

## 7. 前后台与进程重启

### 7.1 前后台

1. 上传中把小程序切到后台。
2. 预期停止调度新的分片请求；已发请求的结果按真实返回处理。
3. 回到前台，先拉取服务端详情，再仅上传 `pending` 分片。
4. 重复两次，确认没有重复定时器、重复队列或进度回退。

### 7.2 临时路径仍有效

1. 至少完成首片后杀掉小程序进程。
2. 保留微信临时文件路径，再打开小程序。
3. 预期读取原文件并逐片核对已上传 SHA-256，继续同一 upload，只传缺失分片。

### 7.3 临时路径失效或源文件变化

1. 至少完成首片后杀进程，并让临时路径失效；另做一次内容变化但路径仍可读的测试。
2. 重新打开。
3. 预期以 `reason=replaced` 中止旧上传，不能把新文件拼接到旧会话。
4. 用户重新选择、二次确认后创建一个新的上传会话。

## 8. finalizer 对账与新的 complete 周期

只在可故障注入的测试环境执行：

1. 上传全部分片，在 complete 响应返回客户端前中断客户端连接。预期服务端保持 `completing/finalizing`，后台通过 R2 事实最终收口；客户端轮询期间不得生成新 complete Key。
2. 注入一次“R2 已完成但 API 响应未知”，预期 HEAD 对账到对象后状态变为 `uploaded`，没有第二个 multipart。
3. 注入一次可修复的 parts 不一致，使服务端明确回到 `uploading` 并把需要修复的 parts 重置为 `pending`。客户端仅重传这些 parts。
4. 只有收到上述明确修复状态后，删除旧 complete-cycle Key；重新完成时生成恰好一个新 UUIDv7 Key，并在该周期所有重试中复用。
5. 检查审计事件与最终历史，状态链可解释且没有伪造成功。

| 场景              | 预期                                | 结果/证据 |
| ----------------- | ----------------------------------- | --------- |
| complete 响应丢失 | 原 Key + 轮询，最终安全收口         |           |
| R2 成功/API 未知  | 对账为 `uploaded`，无重复 multipart |           |
| parts 修复        | 只重传服务端重置的 part             |           |
| 新 complete 周期  | 仅明确修复后产生一个新 Key          |           |

## 9. 多文件部分失败

1. 一次选择 3 个合法文件，其中一个在上传中断网或注入可重试失败。
2. 确认其他文件可独立完成，失败文件有明确状态，不拖成整批假成功。
3. 对失败文件重试，确认复用其正确会话/part；已完成文件不重传。
4. 再测试一个不可重试的格式签名失败，确认其他队列项仍能继续。
5. 历史页刷新、下拉和分页后状态不回退、不重复，也不会出现其他测试账号的数据。

## 10. 私有存储验证

由具有测试 bucket 审计权限的运维人员执行：

1. 确认 bucket 匿名访问关闭，未配置对外对象交付域名。
2. 检查对象键由服务端生成，形如 `users/<内部userId>/<kind>/<year>/<month>/<mediaId>.<ext>`，不同用户前缀隔离。
3. 抓取小程序网络响应，确认没有 `r2UploadId`、object key、bucket 名、S3 endpoint、存储凭据、ETag 或对象地址。
4. 用第二个测试账号请求第一个账号的 uploadId，预期统一 `404`。
5. 对一个未完成 multipart 验证 7 天终止生命周期规则存在；确认完成对象没有自动删除规则。
6. 检查小程序包与 API/Nginx 日志，确认没有 AppSecret、R2 secret、JWT 私钥、数据库密码、昵称或微信底层身份标识。

## 11. 发布签字

以下任一项失败都不得发布：

- 真实登录与稳定内部用户 ID；
- 昵称授权、取消和独立确认语义；
- 11 bytes、200 MiB、200 MiB + 1 byte 边界；
- 全部白名单格式和 magic-byte 反例；
- 二次确认取消不产生上传；
- 网络切换、弱网、离线、15 分钟令牌跨越；
- 前后台与两类进程重启恢复；
- finalizer 对账和新 complete 周期；
- 多文件部分失败；
- 私有 bucket、用户隔离、响应与日志无存储内部信息。
